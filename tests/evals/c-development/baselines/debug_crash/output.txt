A segfault on Windows but not Linux usually points to one of a few
platform-specific differences. Here's the diagnosis order I'd follow:

1. Run with AddressSanitizer (ASAN) first — it catches most memory errors:
   cmake -B out -DMY_UTILS_ENABLE_ASAN=ON
   cmake --build out --config Debug
   ctest --test-dir out -C Debug --output-on-failure

2. If ASAN is clean, check the `long` type. On Windows x64, `long` is 4 bytes;
   on Linux x64 it's 8 bytes. If your code stores a pointer or 64-bit value in
   a `long`, it silently truncates on Windows. Fix: use int32_t or int64_t from
   <stdint.h> instead of `long`.

3. Check for stack overflow. Windows default stack size is 1 MB, Linux is 8 MB.
   Deep recursion or large stack allocations that fit on Linux may overflow on
   Windows. Reduce recursion depth or move large buffers to the heap.

4. If none of the above, attach a debugger. On Windows use Visual Studio or CDB
   (k for backtrace). On Linux use GDB (bt for backtrace).
